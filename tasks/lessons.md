# lessons

- when the user says a project is for a hackathon, do not frame it as a disposable hackathon project by default. treat the hackathon as a deadline, launch surface, and sponsor-alignment constraint unless the user explicitly says it is throwaway.
- product ui copy should use sentence case unless a specific brand or technical token requires different casing. do not let the assistant's lowercase chat style leak into interface labels, buttons, empty states, or dashboard text.
- when the user says production product, agi, or funded-tier, do not shrink the ambition into "just a demo" or "just a vertical slice." use demo slices only as proof checkpoints for a larger production architecture, with durability, trust boundaries, observability, security, multi-tenant scale, and operator controls treated as first-class requirements.
- for showcase pages, lead with the magical interaction instead of static proof cards. metrics should feel like a reveal after the user acts, not a dashboard wall before the product has done anything.
- when the user asks for artifacts, treat that as a visible sidecar surface like claude code/claude artifacts: chat or operator work on one side, rendered artifact preview on the other, with tabs for preview, data, and proof.
